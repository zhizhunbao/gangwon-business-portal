# 产品经理技能使用指南

## 🎯 技能概述

产品经理技能专门处理客户需求分析，特别是客户不懂技术、只能通过PPT表达需求的情况。帮助产品经理将客户的可视化需求转化为可执行的技术方案。

## 🚀 适用场景

### 典型场景
- **客户不懂技术**: 客户只能描述业务流程，无法表达技术需求
- **PPT需求**: 客户提供PPT或设计稿作为需求文档
- **界面导向**: 客户重点关注界面设计和用户体验
- **业务驱动**: 客户关注业务价值和投资回报

### 具体情况
- 客户说："我想要一个像这个PPT一样的系统"
- 客户说："这个功能要实现，那个功能也要有"
- 客户说："界面要好看，操作要简单"
- 客户说："我要这个功能，但不知道技术怎么实现"

## 🔄 使用流程

### 阶段1: 需求接收
```bash
# 接收客户PPT，开始分析
"windsurf: use product-management skill to analyze customer PPT requirements"
```

**具体步骤**:
1. 接收客户PPT文件
2. 使用PPT分析工具自动提取关键信息
3. 填写需求分析报告
4. 识别核心功能和支持功能

### 阶段2: 需求澄清
```bash
# 澄清客户需求，准备原型设计
"windsurf: use product-management skill to clarify customer requirements and design prototype"
```

**具体步骤**:
1. 列出需要澄清的问题
2. 准备客户沟通话术
3. 安排客户沟通会议
4. 记录客户反馈

### 阶段3: 原型设计
```bash
# 创建原型与客户确认
"windsurf: use product-management skill to create interactive prototype for customer confirmation"
```

**具体步骤**:
1. 选择合适的原型工具
2. 创建低保真原型
3. 模拟业务流程
4. 准备原型演示

### 阶段4: 需求确认
```bash
# 与客户确认需求和原型
"windsurf: use product-management skill to confirm requirements with customer"
```

**具体步骤**:
1. 展示原型设计
2. 解释技术方案
3. 确认功能需求
4. 确定项目范围

### 阶段5: 技术转化
```bash
# 将确认的需求转化为技术文档
"windsurf: use product-management skill to convert confirmed requirements to technical specifications"
```

**具体步骤**:
1. 编写详细需求文档
2. 创建技术架构设计
3. 制定开发计划
4. 评估项目风险

## 🎯 最佳实践

### 与客户沟通
- **使用业务语言**: 避免技术术语
- **可视化优先**: 多用原型和图表
- **耐心解释**: 反复确认理解
- **管理期望**: 合理设定预期

### 需求分析
- **结构化分析**: 按页面和功能分类
- **优先级排序**: 区分核心和支持功能
- **可行性评估**: 考虑技术限制
- **风险识别**: 提前发现潜在问题

### 原型设计
- **忠于原PPT**: 保持客户设计意图
- **简化复杂**: 避免过于复杂的设计
- **交互友好**: 确保操作简单明了
- **快速迭代**: 快速制作和修改

### 技术转化
- **详细文档**: 确保技术团队理解需求
- **架构合理**: 设计可扩展的系统架构
- **计划可行**: 制定现实的开发计划
- **风险控制**: 准备风险应对措施

## ⚠️ 常见问题

### Q: 客户要求的功能技术实现不了怎么办？
A: 诚实地说明技术限制，提供替代方案，管理客户期望。

### Q: 客户反复修改需求怎么办？
A: 设置合理的修改次数限制，明确变更成本，建立变更控制流程。

### Q: 客户不理解技术方案怎么办？
A: 使用类比和比喻，重点讲业务价值，避免技术细节。

### Q: 原型与最终产品差异很大怎么办？
A: 在原型阶段明确说明技术限制，分阶段实现，管理客户期望。
