# PRD 编写规范 (江原道企业门户专项)

## 触发场景

- 定义新功能或重构现有模块（如：企业管理、项目申报、公告系统）
- 编写用户故事（User Stories）和验收标准（Acceptance Criteria）
- 评估系统非功能性需求（安全性、多语言兼容性、性能）

## 前置依赖

> **⚠️ 开始编写前，必须先检查以下 skill：**
>
> 1. **`dev-terminology`** - 确保功能命名符合项目术语字典（如：`Enterprise` vs `Company`）
> 2. **`dev-tdd_workflow`** - 确保验收标准（AC）可转化为测试用例

## 核心原则

### 1. 术语一致性 (Terminology)

需求文档中的所有业务对象、状态、角色命名必须严格遵守 **`dev-terminology`**。

- 例如：项目状态应统一使用 `Pending`, `Approved`, `Rejected` 等。
- 企业角色应统一使用 `Member`（企业端）和 `Admin`（管理员端）。

### 2. UI/UX 体验设计 (User Experience)

PRD 必须定义符合 **`dev-frontend_patterns`** 的用户交互流程：

- **专业办公风**：明确要求使用纯白背景、细边框、高透明度卡片布局，保持商务感。
- **状态反馈**：定义提交、审核中、审核通过、驳回等状态的视觉区分。
- **一致性**：所有按钮及表单组件必须优先复用 `frontend/src/shared/components` 中的标准库。

### 3. 多端模块隔离 (Architecture)

在定义需求时，必须明确该功能属于 `Member` 端还是 `Admin` 端：

- **Member (会员端)**：关注企业注册、项目申报进度、咨询反馈。
- **Admin (管理端)**：关注成员审核、申报管理、数据统计、系统日志。

### 4. 国际化 (i18n)

由于是韩国地方政府相关项目，所有功能点必须明确处理**中文**和**韩语**的支持。

- 默认语言环境处理：所有的硬编码文本必须有对应的 `zh.json` 和 `ko.json` 键值。

### 5. 可测性 (Testability)

每一个验收标准（Acceptance Criteria）必须是可验证的。

- ❌ 错误：审核流程要顺畅。
- ✅ 正确：管理员点击“批准”后，系统应在 1s 内更新项目状态为 `Approved`，并向企业发送系统通知。

## PRD 标准结构

### 1. 执行摘要与目标

- 产品/功能名称与版本
- 核心业务痛点：解决了企业申报的什么问题？
- 成功指标：如 流程完成率、审核响应时间。

### 2. 用户故事 (User Stories)

采用标准格式：

> **作为** [角色], **我想要** [动作], **以便** [收益]。

### 3. 功能需求 (Functional Requirements)

- **业务逻辑**：工作流转化、权限校验规则。
- **UI/UX 逻辑**：组件级的功能描述。
- **数据结构**：涉及到的关键字段及其映射。

### 4. 验收标准 (Acceptance Criteria) - 核心

必须包含：

- **功能路径**：用户操作的成功路径。
- **业务校验**：如 必填项检查、重复申报检测。
- **美学校验**：是否符合后台系统设计规范。
- **异常处理**：网络中断、权限不足、数据加载失败。

### 5. 非功能需求 (Non-Functional Requirements)

- **安全性**：敏感数据（如企业税号）的脱敏显示。
- **兼容性**：支持韩语输入法下的文本编辑。

## 模板示例

### 用户故事与 AC 示例

```markdown
#### US-001: 企业入驻申请

**作为** 潜在企业用户，  
**我想要** 填写企业基本信息并上传执照，  
**以便** 获得入驻门户并申报项目的权限。

**验收标准**:

1. [ ] 用户提交申请后，系统生成 `Pending` 状态的审核记录。
2. [ ] 所有的必填字段（企业名称、税号、法人）缺失时应弹出红色 Alert 提示。
3. [ ] 审核流程应在 `Admin` 端同步可见。
4. [ ] 支持中文和韩语界面的字段实时切换。
```

## 维护与更新

- 所有的 PRD 变更必须通过 Git 进行版本控制。
- 需求变更后，必须同步更新对应的 `dev-tdd_workflow` 下的测试用例。
