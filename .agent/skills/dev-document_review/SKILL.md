# 文档评审规范 (江原道企业门户专项)

## 目标

通过一致性、准确性、清晰性、精简度、错误检测、组织结构及内容相关性 7 个维度评审技术文档。

**核心原则：简单就是美 (Simplicity is Beauty)**

- **如无必要，勿增内容**：如果读者不看某部分也能完成任务，就删掉它。
- **一处定义，处处引用**：避免在多个文档中重复解释逻辑。

## 评审维度

### 1. 一致性 (Consistency)

- **术语一致**：是否统一使用 `Enterprise` 而非混用 `Company`？
- **i18n Key**：文档中的翻译键是否与 `locales/*.json` 中的完全一致？

### 2. 准确性 (Accuracy)

- **业务逻辑**：如“申报流程”是否与代码中的状态机逻辑对齐？
- **代码片段**：示例代码是否能在当前技术栈（React 18 / FastAPI）下运行？
- **路径检查**：链接到的 `docs/01-requirements` 等路径是否真实有效？

### 3. 精简度 (Conciseness)

- **拒绝冗余**：不要在 PRD 里重复写 README 已经写过的技术栈。
- **效率优先**：快速让开发者找到 API 的 Base URL 和 Auth Header。

### 4. 内容相关性 (Relevance)

- **模块聚焦**：Admin 端的文档不应包含过多 Member 端的 UI 细节。

---

## 精简指南 (Pruning Guidelines)

当发现以下情况时，建议执行“剪枝”：

1. **重复解释**：多个 Feature PRD 都在解释同一个“文件上传组件”。应汇总至 `docs/02-architecture/standards/`。
2. **过期逻辑**：保留的是最终版本，删除 `legacy/` 目录中已被新方案覆盖的草稿。
3. **过细颗粒度**：不要在文档里描述琐碎的 CSS 像素值，代码是更好的说明。

## 常见问题与修正示例

| 问题类型   | 错误示例                                                  | 修正建议                                             |
| ---------- | --------------------------------------------------------- | ---------------------------------------------------- |
| 术语不一致 | "公司信息" vs "企业资料"                                  | 统一使用 `glossary.md` 中的“企业 (Enterprise)”       |
| 代码不匹配 | 文档显示 `status: "open"`，实际代码为 `status: "PENDING"` | 更新文档以匹配后端枚举值                             |
| 路径失效   | `[架构文档](../architecture.md)`                          | 修正为 `[架构文档](../../02-architecture/README.md)` |
| 冗余描述   | 重复解释如何安装 `npm`                                    | 统引至 `docs/04-development/setup.md`                |

---

## 评审流程

1. **快速扫描**：检查 TOC 和标题是否符合 01-10 目录规范。
2. **一致性检查**：对比代码中的字段名与文档中的 Schema 分类。
3. **精简操作**：删除与核心目标无关的长篇大论。
4. **生成报告**：指出需要修正的具体行号和改进方案。
